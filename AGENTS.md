# JMComic-Crawler-Python 项目规则

## 基本原则

1. 这是一个 Python 库 + CLI 项目，所有改动都要优先保持原有下载逻辑、命令行入口和对外 API 兼容。
2. 新功能优先采用“新增模块、 新增接口、 新增入口、 新增目录”的方式实现，避免推翻原有架构。
3. 除非需求明确要求，否则不要修改 `src/jmcomic` 中现有的下载核心逻辑。
4. 所有新增或修改的文件必须使用 UTF-8 编码且不带 BOM。
5. 默认使用中文回复；不确定的地方必须明确说明“不确定”，不要猜测。

## 开始任务前

1. 每次开始任务前，先识别当前 workspace / repository / project root。
2. 每次开始任务前，优先读取当前项目根目录的 `AGENTS.md`。
3. 如果任务涉及更深层目录，再继续读取更近层级的 `AGENTS.md`。
4. 如果仓库没有 `AGENTS.md`，先说明缺失，再决定是否补充，不要默认沿用别的项目规则。

## 工作流程

1. 完整功能、跨文件修改、重构、删除代码、跨模块联调，默认先进入计划阶段，只读分析，不直接改业务代码。
2. 计划阶段要说明：
   - 当前识别到的项目名称
   - 当前项目技术栈
   - 已读取的 `AGENTS.md` 和关键文件
   - 需求理解
   - 涉及文件
   - 具体步骤
   - 风险点
   - 验证方式
3. 未确认计划前，不要贸然开始大范围改动。
4. 小范围明确修复也必须先阅读相关文件，不能凭印象写代码。
5. 修改完成后要总结：
   - 修改了哪些文件
   - 为什么这样改
   - 是否影响现有功能
   - 运行了哪些验证命令
   - 是否还有风险

## 变更边界

1. 不要顺手重构无关文件。
2. 不要格式化无关文件。
3. 不要扩大需求边界。
4. 不要引入新依赖，除非先说明原因、影响、替代方案，并得到确认。
5. 删除代码前必须先证明未被使用；无法确认无用时，只能标记为“疑似无用”。
6. 大范围重构必须先给出计划和回滚思路，且保持业务行为不变。

## Python 代码约定

1. 优先复用现有的模块、函数、类和命名风格。
2. 保持 CLI 继续可用，不要破坏现有脚本入口。
3. 新增 Web / UI / 工具类能力时，优先做成独立模块或独立包，再通过现有 API 连接核心逻辑。
4. 涉及配置、路径、缓存、日志、数据库时，尽量可配置，默认值要安全可用。
5. 新增代码要带清晰注释，尤其是异步任务、持久化、状态流转和跨模块桥接处。

## 项目特定约定

1. 这个仓库当前以“下载漫画核心能力”为主，任何管理界面、任务队列、历史记录、封面预览、翻译等能力，都应作为增量功能接入。
2. 如果需要做可视化界面，优先使用轻量方案，避免引入复杂前端构建链。
3. 如果需要持久化，优先使用本地可嵌入方案，保证开包即用、重启可恢复。
4. 所有新入口、模板、静态资源、数据库文件路径，都应尽量支持配置化。
